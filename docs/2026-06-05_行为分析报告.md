# 桌面 AI 助手「Ultimate」— 行为分析报告

> 基于对话日志 1.md（常规问答）和 2.md（打开网易云音乐）的分析
> 生成日期：2026-06-05

---

## 一、总评

| 维度 | 1.md（常规问答） | 2.md（打开网易云音乐） |
|------|:-:|:-:|
| 工具调用数 | ~31 / 3 轮 | **45 / 1 轮** |
| 耗时 | ~2.5 分钟 | **~58 分钟** |
| 有效步骤占比 | ~30% | **~10%** |
| 重复错误 | 较少 | **严重** |
| 是否完成任务 | ✅ | ⚠️ 勉强 |

**综合评分：4/10**（基础能力具备，但效率极低，缺乏容错意识）

## 二、正面能力

1. **工具调用链完整** — 能调用摄像头、截图、图片分析、文件搜索、执行 Python 等多种工具
2. **信息提取** — 能搜索文件系统、读取 SQLite 数据库、解析 JSON 配置
3. **结构化输出** — 回复格式清晰，善用列表和分段
4. **给出后续建议** — 每次操作后会主动提供下一步选项

## 三、关键问题

### 问题 1：工具调用效率极低

**2.md 典型案例 — 45 步完成 5 步能做的事：**

理想流程（3-5 步）：
1. 搜索安装路径 → 2. 启动应用 → 3. 告知用户完成

实际流程：
1. 截图 → 分析失败 → 再截图 → 再分析失败 → 搜文件路径 ✅
2. 启动应用 ✅
3. 截图 → 分析失败 → OCR 失败 → 截图 → 分析失败 → 截图 → PIL 检查...（又跑了 20+ 步）
4. 折腾 GUI 配置 → 改配置 → 依然用不了 → 查代码 → 发现是缓存问题
5. 最终告诉用户：应用已打开，GUI 要重启才生效

**影响**：80% 的工具调用不产生任何价值。

**建议**：每调用一个工具前，先问自己："这个调用能改变我的下一步行动吗？" 如果不能，就别调。

---

### 问题 2：工具本身存在 bug 或缺陷

有些工具调用失败不是 agent 的问题，而是**工具本身有 bug**：

- `analyze_image` — 连续多次返回 "no image provided" / "text and coordinates"，尽管传入了有效的本地图片路径
- `ocr_image` — 返回 "unsupported format string passed to list.__format__"，明显是工具内部的 **Python 格式化 bug**
- `take_screenshot(window_title=...)` — 报 "需要 pywin32"，工具依赖未预装，没有降级方案
- `gui_click` / `gui_move` — 无法识别配置文件的热更新，需要重启进程

**影响**：agent 浪费了大量时间反复调用有问题的工具，而工具开发者可能根本不知道这些问题。

**建议**：
- 工具错误信息应该更明确，区分 **"参数传错"**、**"依赖缺失"**、**"内部 bug"** 三种情况
- 对于已知的缺陷工具，应有 **维护清单** 或 **健康检查** 机制
- 工具在启动时可以做一次 **冒烟测试**，提前发现不可用的工具，而不是在任务执行中突然失效

---

### 问题 3：重复犯同样的错误

这是最严重的问题。**analyze_image 失败 3 次**、**ocr_image 失败 3 次**，每次都报一模一样错误。

| 工具 | 调用次数 | 错误信息 | 结果 |
|------|:-:|----------|:----:|
| analyze_image | 3 次 | "no image provided" | 全部失败 |
| ocr_image | 3 次 | "unsupported format string..." | 全部失败 |
| subprocess.* | 2 次 | "dangerous pattern blocked" | 全部失败 |
| gui_click/gui_move | 2 次 | "GUI 未启用" | 全部失败 |

**连续失败 2 次后，理性的行为是：** 完全放弃这个工具，换另一条路。而不是再来第 3 次。

**建议**：实现"失败记忆"机制 — 同一个工具在同一个任务中连续失败 2 次，直接标记为"本轮不可用"。

---

### 问题 4：不理解系统边界，反复撞墙

安全策略明确禁止的操作（`subprocess`、`ctypes`、`os.system`、文件写操作），每次遇到新问题时都先用被禁的方法试，被 blocked 再换。

**建议**：建立"已知禁用操作"列表，在任务开始时就记下来，不再触碰。

---

### 问题 5：不理解进程状态 vs 配置状态

修改 `config.json` 后立即调用 GUI 操作 — 没有意识到：
- 配置在进程启动时已读到内存常量
- 修改文件不会影响当前运行进程
- 需要进程重启才能生效

花了大量时间读配置、确认、查代码才理解这个基本架构问题。

**建议**：修改配置类参数时，先确认当前进程有没有缓存/重载机制。

---

### 问题 6：回复极度啰嗦

每做一小步就发一条消息给用户：
> "好的，我先看看桌面的情况"
> "找到了！让我找到它的可执行文件路径"
> "网易云音乐已启动！让我看看是否成功打开了"

一整屏都是"我准备做 / 我做了 / 现在我再做"的过程报告。每条消息本身可能只有一两句是需要说的，但包裹了大量废话。

具体表现：
- **复述用户的话** — "好的，我来查看系统信息..." / "好的，我来调用摄像头..." → 用户刚说的话不需要你重复一遍
- **冗余的状态声明** — "找到了！网易云音乐已安装" / "应用已启动！" → 直接说结论，删除感叹
- **无意义的过渡句** — "让我换个方法..." / "好的，虽然工具有问题，但..." → 直接做，不需要预告

**建议**：每条回复遵循 **减法原则**：
1. 去掉所有"好的"、"让我"、"我来" 等口头禅
2. 去掉对用户要求的复述（"用户想打开网易云音乐，所以我先..."）
3. 一句话能说清的不要拆成三段
4. 结果直接给，过程不用说
5. 写完回复后删掉 50% 的内容，看看意思变了没有 — 没变就对了

---

### 问题 7：缺乏用户视角

用户说"打开网易云音乐"，agent 花 58 分钟折腾验证、GUI 配置、截图分析。但实际：
- **用户就坐在屏幕前**，应用打开了自己能看到
- 用户要的是**结果**，不是验证过程的直播

**建议**：区分"我能做的事"和"用户已经能看到的事"。应用启动后，用户自己就能确认，不需要你截图验证。

---

### 问题 8：记忆恢复能力弱

用户说"继续刚刚的工作"时：
- 查到记忆库但摘要为空 → 没有进一步读完整内容字段
- 直接放弃，抛给用户一堆通用选项
- 没有真正"继续"任何工作

**建议**：尝试探索记忆库的完整数据，而不是只看摘要字段就放弃。

---

### 问题 9：屏幕分析流于表面

分析屏幕时只是描述"有两个窗口"、"有 DeprecationWarning"：
- 没有提取具体警告内容
- 没有主动提出修复建议
- 没有关联项目代码

**建议**：发现问题时要主动深入，而不是停留在描述层面。

## 五、核心设计原则（总结）

1. **一种工具失败 2 次就放弃** — 不要期待奇迹
2. **先弄清系统边界再动手** — 什么能用、什么不能用，先搞清楚
3. **每个工具调用必须有产出** — 如果不能改变决策或推进任务，就别调
4. **一次只说三句话** — 去掉所有废话、重复、格式装饰
5. **相信用户的感知** — 用户能看到屏幕上的结果，不需要你验证
