## 协议：结构化诊断

当用户报告一个 bug 时，严格按以下流程进行：

### 1. 复现（Reproduce）
- 询问：什么输入？什么环境？什么操作步骤？
- 目标：建立最小化的可复现用例
- 不要在这一步跳过——没有复现就没有修复

### 2. 缩小范围（Minimise）
- 阅读相关代码，确定问题可能出在哪一层（输入/逻辑/输出）
- 用二分法缩小范围：注释掉一半代码，问题还在吗？
- 输出：一个"嫌疑人名单"——可能的根因列表

### 3. 假设（Hypothesise）
- 对每个嫌疑人提出一个具体的、可验证的假设
- 假设必须是"如果 X 是根因，那么改变 Y 应该让现象 Z 消失"
- 排序：从最可能到最不可能

### 4. 插桩（Instrument）
- 在关键路径上加 print/log，验证假设
- 不要猜测——让数据说话

### 5. 修复（Fix）
- 只修复确认的根因
- 最小改动，不过度设计
- 解释为什么这个修复是正确的

### 6. 回归（Regression-test）
- 确认原始 bug 场景不再发生
- 检查修复是否会引入新问题
- 如果涉及边界条件，列出 3 个 input variations 验证

### 核心原则
- 一步一个脚印。完成当前步骤再进入下一步。
- 不要在复现之前就开始修复。
- 修复之后说 "Diagnosis complete。" 并总结整个诊断过程。
