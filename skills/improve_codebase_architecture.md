## 协议：代码架构改进

找出架构摩擦点并提出**深化机会**——把浅模块重构为深模块。

### 核心概念

- **模块** — 有接口和实现的任何东西（函数、类、包、切片）
- **接口** — 调用者必须知道的全部：类型、不变量、错误模式、顺序、配置（不只是类型签名）
- **深度** — 接口的杠杆率：小接口背后大行为 = **深**。接口几乎跟实现一样复杂 = **浅**
- **接缝** — 接口所在的位置，不原地编辑就能改变行为的地方
- **删除测试** — 想象删除这个模块。如果复杂度消失，它是直通层。如果复杂度扩散到N个调用方，它在创造价值。

### 流程

#### 1. 扫描
浏览代码库，留意以下摩擦点：
- 理解一个概念需要跳转多少个小模块？
- 哪里有浅模块（接口几乎跟实现一样复杂）？
- 哪里为了可测试性提取了纯函数，但真正的 bug 藏在调用方式里（缺乏局部性）？
- 哪里紧耦合模块跨越接缝泄露？
- 哪些部分没有测试，或通过当前接口很难测？

#### 2. 提出候选
对每个候选给出：
- **涉及文件** — 哪些文件/模块
- **痛点** — 当前架构为什么造成摩擦
- **方案** — 改动什么，用自然语言描述
- **收益** — 从局部性和杠杆率角度解释

不要提出具体的接口设计。先问用户："想深入看哪个？"

#### 3. 追问循环
用户选定一个候选后，进入追问模式：走通限制条件、依赖关系、深化后的模块形态、接口后面是什么、哪些测试会保留。

### 原则
- 用领域词汇描述业务，用架构词汇描述代码结构
- 如果候选跟已有 ADR 冲突，只在摩擦确实值得重新审视 ADR 时才提出来
