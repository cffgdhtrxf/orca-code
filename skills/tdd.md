## 协议：测试驱动开发

### 核心理念

测试应通过**公共接口验证行为**，而不是实现细节。代码可以彻底重写，测试不应该因此变动。

**好测试** = 集成风格：通过公共 API 执行真实代码路径。描述系统"做什么"而非"怎么做"。可读性像规格说明书——"用户可以凭有效购物车结账"让你立即知道系统能力。重构时不坏。

**坏测试** = 耦合实现细节：mock 内部依赖、测试私有方法。警告信号：重构时测试坏掉，但行为没变。

### 反模式：水平切片

❌ 不要先写所有测试、再写所有实现。这是"水平切片"——把 RED 阶段当"写全部测试"，GREEN 当"写全部代码"。

这会产生**烂测试**：
- 批量写的测试测试的是**想象的行为**，不是**实际的行为**
- 你在测试**数据结构的形状**，而不是用户可见的行为
- 行为出错时测试通过，行为正确时测试失败

✅ 正确做法：**垂直切片**。一个测试→一个实现→重复。

```
错误（水平）：
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

正确（垂直）：
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
  ...
```

### 流程

#### 1. 规划
- 确认要改什么接口
- 确认要测哪些行为（排优先级）
- 先拿用户批准

#### 2. 曳光弹
写一个测试完成一件事：
```
RED:   写第一个行为的测试 → 测试失败
GREEN: 写最少的代码让它通过 → 测试通过
```
这颗"曳光弹"验证整个路径是通的。

#### 3. 增量循环
对每个剩余行为：
```
RED:   写下一个测试 → 失败
GREEN: 最少的代码让它通过 → 通过
```
规则：
- 一次一个测试
- 只写能通过当前测试的代码
- 不要预判未来的测试
- 测试专注于可观察行为

#### 4. 重构
所有测试通过后检查：
- 消除重复
- 深化模块（把复杂逻辑藏在简单接口后面）
- 每一步重构后跑测试

**永远不要在 RED 时重构。** 先到 GREEN。

### 每次循环检查清单
- [ ] 测试描述行为，而非实现
- [ ] 测试只用公共接口
- [ ] 测试能扛住内部重构
- [ ] 代码对于这个测试是最少的
- [ ] 没有添加推测性功能
